# CanCan — 토스 Product Designer Challenge 제출 답변

> 과제: **6명의 동료가 1시간 회의 일정을 잡는 경험 설계**
> 제출물: https://cancantime.vercel.app  ·  GitHub: https://github.com/seohyeonsong/CanCan
> (핵심 플로우 체험: 로그인 화면 → "데모 계정으로 둘러보기" → 이미 4명이 응답한 회의로 시작합니다)

---

## 1. 어떤 문제를 발견했나요? 왜 문제라고 봤나요?

6명이 시간을 잡을 때 진짜 병목은 **"언제 되세요?" 핑퐁**이 아니라, 그 뒤에 있습니다.

**① 응답하는 사람: "안 되는 시간을 다 칠하는 게" 일이다.**
When2meet류는 되는/안 되는 칸을 다 표시하게 합니다. 6명 각자에게 이건 꽤 큰 노동이고,
칸이 많을수록 대충 찍거나 응답을 미룹니다. **응답 마찰이 곧 응답률**입니다.

**② 주최자: "모아 보여주기"는 있는데 "그래서 언제?"가 없다.**
색칠된 표를 눈으로 훑으며 "필수 참석자는 다 되나, 안 되면 누굴 빼나"를 사람이 판단합니다.
**결정의 부담이 도구가 아니라 사람에게 남아있는 것**이 핵심 문제였습니다.

**③ 시간이 맞아도 "참여 방식"에서 또 어긋난다.**
하이브리드 근무에선 같은 시간이라도 누군 온라인만, 누군 오프라인만 됩니다.
시간만 맞춰놓고 "저는 그날 재택이라 사무실은…"이 뒤늦게 터집니다.

세 문제 모두 **"시간을 모으는 것"과 "회의를 확정하는 것" 사이의 간극**에서 나옵니다.
사용자가 원하는 건 색칠된 표가 아니라 **"이 시간으로 하세요"라는 답**입니다.

---

## 2. 어떻게 해결했나요? (핵심 플로우)

**링크 생성 → 참여자 응답 → 자동 추천 → 확정 → 공유**

**① 응답은 "되는 시간만" — 나머지는 자동 불가.** (가설: 응답 마찰↓)
안 되는 시간을 칠할 필요 없이 **가능한 시간만 드래그**합니다. 표시 안 한 칸은 참석 불가로 처리돼,
6명 각자가 최소 동작으로 응답합니다. 좋아요/괜찮아요/조율 가능 3단계로 뉘앙스만 더합니다.

**② 남의 응답을 겹쳐 보며 응답한다 — "핑퐁" 자체를 제거.**
응답 화면에 앞선 참여자들의 가능 시간이 **테두리 농도(진할수록 여러 명)로 겹쳐** 보여서,
"다들 되는 시간"에 자연스럽게 맞추게 됩니다. 전원이 응답한 뒤엔 **아무도 안 되는 칸은 잠가서**
고를 수 없는 시간에 표시하는 실수를 원천 차단합니다.

**③ 표가 아니라 "추천 시간 카드"로 결정을 거든다.**
응답이 모이면 알고리즘이 모든 후보를 점수화해 **상위 3개를 카드로** 보여줍니다.
**필수 참석자가 빠지는 시간은 후순위**로 밀리고, 카드마다 "가능/불가/조율 필요 N명"이 명시됩니다.
주최자는 표를 해석하지 않고 **카드를 고르기만** 하면 됩니다.

**④ 참여 방식을 시간과 동급으로 받는다.**
참여자는 가능 시간과 함께 **온라인/오프라인/모두 가능**을 선택하고,
추천 카드 이름 옆에 온라인/오프라인 아이콘이 붙어 **확정하는 그 자리에서** 방식까지 판단합니다.

**⑤ 확정에서 안 끝내고 "다음 행동"까지.**
확정 시 **캘린더에 바로 추가**하고, 참여자에게 **공유(공유 시트로 슬랙·카톡 전송)**합니다.
겹치는 시간이 없으면 **"가능한 시간을 더 표시해달라"는 재조율 요청**을, 미응답자에겐 **응답 재촉**을
한 번에 보낼 수 있어, 조율이 막혔을 때의 다음 행동까지 이어집니다.

---

## 3. 왜 이렇게 설계했나요?

**① "모아 보여주기"가 아니라 "추천해주기"로 밀어붙인 이유.**
사용자의 목표는 "가능 시간 확인"이 아니라 "회의 확정"입니다. 그래서 도구가 판단을 대신 해주는
지점(추천 3개 + 가능/불가 명시 + 필수 가중치)까지 갔습니다. 단, **최종 결정권은 사람에게** 남겨
"추천하되 강요하지 않는다"를 지켰습니다.

**② "안 되는 시간 표시"를 없앤 이유.**
복잡한 규칙(각자 되는/안 되는 시간)을 **"되는 것만 칠하면 나머지는 시스템이 처리"**로 뒤집어,
응답자의 인지 부담을 최소화했습니다. 6명이 빠르게 응답할수록 회의는 빨리 잡힙니다.

**③ 참여 방식을 1급 정보로 끌어올린 이유.**
하이브리드는 이미 기본값입니다. 나중에 확인하는 부가 정보가 아니라 시간과 동급으로 설계해
**"시간을 맞춰놓고 방식에서 어긋나는 상황" 자체를 없앴습니다.**

**④ 참여자 로그인 없음 (마찰 최소화).**
초대받은 사람에게 가입을 요구하는 건 응답률을 떨어뜨리는 가장 큰 마찰입니다.
참여자는 링크만으로 응답하고, 주최자만 회의 관리를 위해 로그인합니다.

**⑤ 동명이인 안전장치.**
큰 조직엔 동명이인이 흔합니다. 이미 응답한 이름과 같은 이름으로 응답하려 하면 막고,
"본인이면 '응답 수정', 다른 사람이면 구분 표시(예: 송서현A)"로 안내해 **응답 덮어쓰기를 방지**합니다.

**⑥ 모바일·PC 모두에서 같은 경험.**
시간 그리드는 **터치 드래그·마우스 드래그**를 모두 지원하고, 좁은 폰부터 데스크탑까지 반응형으로 대응합니다.

### 기술 스택
React + TypeScript + Vite / CSS Modules / localStorage(프로토타입 저장)

### 프로토타입 범위 & 확장 지점 (의도적 한계)
핵심 흐름의 **설계·경험 검증**에 집중한 백엔드 없는 프론트엔드 프로토타입입니다.
- **데이터:** 브라우저 localStorage라 기기 간 실시간 동기화는 없습니다. 데모는 4명의 응답을 미리 채워 다중 참여자 경험을 재현합니다. → 실서비스는 DB + 실시간 동기화.
- **알림:** 확정·재촉을 공유 시트(Web Share)로 처리. → 실서비스는 자동 알림(푸시/캘린더 초대/슬랙 봇).
- **인증:** 데모용 단일 계정. → 실서비스는 개인 계정·권한.
