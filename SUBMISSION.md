# CanCan — 토스 Product Designer Challenge 제출 답변

> 과제: **6명의 동료가 1시간 회의 일정을 잡는 경험 설계**
> 제출물: https://cancantime.vercel.app  ·  GitHub: https://github.com/seohyeonsong/CanCan
> (체험: 로그인 화면 → "데모 계정으로 둘러보기" → 이미 4명이 응답한 회의로 시작합니다)

---

## 1. 어떤 문제를 발견했나요? 왜 문제라고 봤나요?

6명이 시간을 잡을 때 진짜 병목은 "언제 되세요?" 핑퐁이 아니라 **그 앞뒤에** 있습니다.

**① 응답하는 사람 — "안 되는 시간까지 다 칠하는 게" 일이다.**
기존 도구(When2meet류)는 되는/안 되는 칸을 다 표시하게 합니다. 6명 각자에게 이건 은근한 노동이고,
칸이 많을수록 대충 찍거나 응답을 미룹니다. **응답 마찰이 곧 응답률**이고, 회의는 응답이 다 모여야 잡힙니다.

**② 주최자 — "모아 보여주기"는 있는데 "그래서 언제?"가 없다.**
색칠된 표를 눈으로 훑으며 "필수 참석자는 다 되나, 안 되면 누굴 빼나"를 사람이 판단합니다.
**결정의 부담이 도구가 아니라 사람에게 남아있는 것**이 핵심 문제였습니다.

**③ 시간이 맞아도 "참여 방식"에서 또 어긋난다.**
하이브리드 근무에선 같은 시간이라도 누군 온라인만, 누군 오프라인만 됩니다.
시간만 맞춰놓고 "저는 그날 재택이라 사무실은…"이 뒤늦게 터집니다.

세 문제 모두 **"시간을 모으는 것"과 "회의를 확정하는 것" 사이의 간극**에서 나옵니다.
사용자가 원하는 건 색칠된 표가 아니라 **"이 시간으로 하세요"라는 답**입니다.

---

## 2. 어떻게 해결했나요? (핵심 플로우)

**만들기 → (주최자 본인 응답) → 참여자 응답 → 자동 추천 → 확정 → 공유**

**① 응답을 "예쁘게"가 아니라 "적게" 만들었다.** (문제 ①)
- 안 되는 시간을 칠할 필요 없이 **가능한 시간만** 표시 — 나머지는 자동으로 불가 처리.
- 응답 화면에 앞선 참여자들의 가능 시간을 **테두리 농도(진할수록 여러 명)로 겹쳐** 보여줘, "다들 되는 시간"에 자연스럽게 맞추게 했습니다.
- 결정적으로 **「다들 가능한 시간 먼저 채우기」** 한 번이면 겹치는 시간이 내 응답으로 미리 채워지고, **안 되는 것만 빼면** 됩니다. "빈 표 채우기"를 "예외만 빼기"로 뒤집은 것.
- 전원이 응답한 뒤엔 **아무도 안 되는 칸은 잠가서** 헛클릭을 막습니다.

**② 표가 아니라 "추천 카드"로 결정을 거든다.** (문제 ②)
- 응답이 모이면 알고리즘이 후보를 점수화해 **상위 3개를 카드로** 제시. **필수 참석자가 빠지는 시간은 후순위**로 밀립니다.
- 카드마다 "가능/불가/조율 필요"가 명시되고, **필수 참석자는 이름 앞 파란 점**으로 구분돼 "필수는 다 되고 선택만 빠졌으니 확정해도 되겠다"가 한눈에 읽힙니다.
- 불가한 사람이 있으면 그냥 확정하지 않고 **「○○님에게 조율 요청」**으로 그 사람에게만 시간 조정을 물을 수 있습니다.

**③ 참여 방식을 시간과 함께, 의식적으로 받는다.** (문제 ③)
- 참여자는 가능 시간과 함께 **온라인/오프라인/모두 가능**을 고르고, 추천 카드에 참여 방식 아이콘이 붙습니다.
- 기본값을 두지 않아, 안 고르고 제출하려 하면 **해당 영역으로 스크롤·강조**해 반드시 한 번 선택하게 했습니다.

**④ 확정에서 끝내지 않고 "다음 행동"까지.**
- 주최자는 **회의를 만든 직후 본인 가능 시간을 바로 입력**합니다(정작 중요한 주최자 시간이 누락되던 문제). 참석 안 하면 건너뛸 수 있습니다.
- 확정 시 **캘린더에 바로 추가**하고 참여자에게 **공유(공유 시트로 슬랙·카톡 전송)**. 겹치는 시간이 없거나 미응답자가 있으면 **재조율 요청 / 응답 재촉**을 한 번에 보냅니다.

---

## 3. 왜 이렇게 설계했나요?

**① "모아 보여주기"가 아니라 "추천"까지 간 이유** — 사용자 목표는 "가능 시간 확인"이 아니라 "회의 확정"입니다. 그래서 도구가 판단을 거드는 지점(추천 3개 + 필수 가중치 + 가능/불가 명시)까지 밀되, **최종 결정권은 사람**에게 남겨 "추천하되 강요하지 않는다"를 지켰습니다.

**② "안 되는 시간 표시"를 없앤 이유** — 각자 되는/안 되는 시간이라는 복잡한 규칙을 **"되는 것만, 그마저도 남들 겹침에서 빼기"**로 단순화해 인지 부담을 최소화했습니다. 6명이 빨리 응답할수록 회의는 빨리 잡힙니다.

**③ 참여 방식을 1급 정보로 올린 이유** — 하이브리드는 기본값이라, 나중에 확인하는 부가 정보가 아니라 시간과 동급으로 설계해 "맞춰놓고 어긋나는 상황"을 없앴습니다.

**④ 마찰 제거** — 참여자는 로그인 없이 링크만으로 응답합니다. 또 큰 조직엔 동명이인이 흔해, 이미 응답한 이름으로 다시 응답하려 하면 막고 구분 표시를 유도해 **응답 덮어쓰기**를 방지했습니다.

**⑤ 아직 안 푼 근본 문제 (정직한 한계)** — 이 문제의 가장 깊은 해결은 **"응답 자체를 없애는 것"**, 즉 캘린더를 연동해 비는 시간을 자동 제안하는 것입니다. 이번엔 "손으로 칠하는 응답"을 최대한 가볍게 만드는 데 집중했고, **캘린더 API 연동을 다음 스텝**으로 봅니다.

---

### 기술 스택 / 프로토타입 범위
React + TypeScript + Vite / CSS Modules / localStorage. 핵심 흐름의 **설계·경험 검증**에 집중한 백엔드 없는 프론트엔드 프로토타입입니다.
- **데이터:** localStorage라 기기 간 실시간 동기화는 없습니다. 데모는 응답 4건을 미리 채워 다중 참여자 경험을 재현합니다. → 실서비스는 DB + 실시간 동기화 + 캘린더 연동.
- **알림:** 확정·재촉·조율을 공유 시트(Web Share)로 처리 → 실서비스는 자동 알림(푸시/캘린더 초대/슬랙 봇).
- **인증:** 데모용 단일 계정 → 실서비스는 개인 계정·권한.
